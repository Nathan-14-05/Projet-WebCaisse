Analyse de la faille (DICP & RGPD) :

Confidentialité : L'envoi de mots de passe en clair par e-mail expose les accès à une interception réseau et à un stockage non sécurisé dans les boîtes mail des utilisateurs.

Conformité RGPD : Cette pratique contrevient aux obligations de protection des données personnelles, exposant l'entreprise à des sanctions en cas de fuite d'accès.

Processus sécurisé proposé :

Génération de jeton : Lors de l'inscription, le compte est créé sans mot de passe définitif.

Lien unique : Le système envoie par e-mail un lien temporaire, unique et éphémère (token) menant vers une page de définition du mot de passe.

Hachage sécurisé : Lors de la définition, le mot de passe est immédiatement haché côté serveur via un algorithme robuste (ex: bcrypt ou argon2) avant stockage, garantissant qu'aucun mot de passe en clair ne circule ni ne soit enregistré.

**Question 1.3**
====
La structure proposée par mon collègue est inadaptée pour gérer l'historique de facturation pour les raisons suivantes :

Absence de temporalité (Dimension temps) : La table actuelle ne contient aucune colonne permettant de dater l'activation d'une formule (ex: dateDebut, dateFin). Sans ces informations, il est impossible de savoir quelle formule était active à quel moment précis du mois.

Impossibilité de calcul de facturation : Pour facturer au prorata temporis (selon la durée d'activation), le système doit connaître la durée exacte entre deux changements de formule. La structure actuelle ne permet pas de reconstruire cette chronologie.

Clé primaire : (idPointDeVente, dateDebut).

Avantages :

CREATE TRIGGER tg_audit_changement_formule
AFTER UPDATE ON PointDeVente
FOR EACH ROW
BEGIN
    IF OLD.idFormule <> NEW.idFormule THEN
        INSERT INTO AuditFormule (idPointDeVente, ancien_idFormule, nouveau_idFormule, date_changement)

Le trigger s'active après chaque UPDATE sur PointDeVente.

La condition IF vérifie qu'il y a bien un changement de formule.

L'instruction INSERT archive les données dans AuditFormule.

== QUESTION DE SYNTHESE

====
1) Réglementation NF525
====
La norme NF525 impose des critères d'inaltérabilité, de sécurisation, de conservation et d'archivage des données de transaction. Pour WebCaisse, cela signifie :

Toute vente doit être enregistrée sans possibilité de modification ou de suppression.

Il faut générer une signature numérique pour chaque ticket.

Il faut maintenir une piste d'audit pour prouver que les données n'ont pas été manipulées.

====
2) Identifiants en clair
====
C'est une faille critique car une messagerie (SMTP/IMAP) n'est pas toujours sécurisée. Un attaquant pratiquant une attaque "Man-in-the-Middle" peut intercepter les paquets et lire les identifiants. De plus, les logs des serveurs de mail conservent ces informations, exposant le système à un vol de compte permanent.

====
3) Historique de FormuleSouscrite
====
La structure initiale ne contenait probablement pas de date de début et de date de fin (ou une clé primaire composée avec une date). Sans colonne temporelle, toute nouvelle souscription écrase l'ancienne (UPDATE au lieu d'un INSERT), rendant impossible la consultation de ce que le client payait le mois précédent.

====
4) Intérêt d'un Trigger pour la traçabilité
====
Le Trigger s'exécute automatiquement côté serveur de base de données. Son intérêt est l'indépendance vis-à-vis de l'application : même si un administrateur modifie la base de données manuellement via un outil SQL, le Trigger enregistrera quand même la modification dans une table d'audit, garantissant une traçabilité inviolable par le code applicatif.

====
5) WHERE et HAVING
====
WHERE : Filtre les lignes avant le regroupement (Group By). On l'utilise pour filtrer les données brutes (ex: WHERE montant > 10).

HAVING : Filtre les groupes après le regroupement. On l'utilise pour filtrer sur des résultats d'agrégats (ex: HAVING COUNT(id) > 5).

====
6) Injection SQL par concaténation
====
L'injection survient quand on construit la requête ainsi : "SELECT * FROM users WHERE login = '" + user + "'".
Si un utilisateur malveillant saisit ' OR '1'='1, la requête devient : SELECT * FROM users WHERE login = '' OR '1'='1'. La clause devient toujours vraie, contournant ainsi toute authentification.

====
7) PreparedStatement et Statement
====
Sécurité : Le PreparedStatement utilise des paramètres liés (?), ce qui empêche l'injection SQL car les données sont traitées comme des valeurs et non comme du code SQL.

Performance : La requête est pré-compilée par la base de données. Si elle est exécutée plusieurs fois avec des valeurs différentes, elle est beaucoup plus rapide qu'un Statement classique.

====
8) Utilisation de instanceof
====
Intérêt : Il permet de vérifier dynamiquement le type d'un objet (ex: if (vente instanceof VenteInternet)) pour exécuter un code spécifique.

Critique : Son usage est souvent considéré comme un "code smell" (mauvaise pratique). Il casse le polymorphisme. Il est préférable d'utiliser des méthodes abstraites dans la classe mère et de redéfinir le comportement dans chaque classe enfant.

====
9) Division par zéro
====
En Java, diviser un entier par zéro lève une ArithmeticException qui fait planter le thread. Dans le calcul d'un indice de comparaison, si le dénominateur est nul (ex: aucune vente sur la période de référence), le programme s'arrête. Il est crucial de tester si le dénominateur est != 0 pour éviter le crash et afficher un résultat cohérent (ex: "Indice non calculable").

====
10) Rôle de Launcher et Main
====
Main : Hérite de Application et contient le code JavaFX (chargement du FXML, création de la Stage).

Launcher : Il contient uniquement la méthode public static void main qui appelle Main.main.

Pourquoi ? Depuis Java 11+, JavaFX est devenu un module externe. Cette séparation évite des erreurs de type "JavaFX runtime components are missing" lors du lancement, car le Launcher démarre en mode "standard" avant de basculer dans le contexte JavaFX.
